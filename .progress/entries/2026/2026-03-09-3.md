## Title
Prefer successful ports in known-first scans

## Background / Issue
Real CLI output showed that the cached-port implementation still treated every historically seen port as a first-stage candidate. That contradicted the intended behavior: stage one should only probe ports that had succeeded before, while keeping a broader history of all seen ports for future bookkeeping.

## Actions / Outcome
- Approach 1: Reuse the generic scanned-port cache for known-first -> fast to implement but caused repeated probing of historically failing ports.
- Approach 2: Add regression tests around successful-port semantics and CLI fallback ordering -> exposed that the cache needed a success dimension, not just a seen-port dimension.
- Final approach: Keep `scanned_ports` as the history table, add `last_success_at`, query only successful ports for stage one, and mark successful probe results after each run -> aligns behavior with the intended two-stage scan flow.

## Lessons / Refinements
- Discovery history and successful-candidate history are related but not equivalent; they need separate query semantics even if they share one table.
- Backward-compatible runtime DB changes should include explicit schema migration for existing local databases.
