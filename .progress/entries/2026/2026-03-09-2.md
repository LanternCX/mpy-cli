## Title
Avoid duplicate probes in known-first fallback

## Background / Issue
After introducing cached-port scanning for `mpy-cli list`, real CLI output showed two regressions: an explicit empty first-stage port list silently fell back to `connect list` inside backend probing, and `known-first` fallback re-probed ports that had already failed in stage one.

## Actions / Outcome
- Approach 1: Inspect CLI logs only -> confirmed duplicate probing but did not isolate which layer caused it.
- Approach 2: Add regression tests for explicit empty port lists and no-repeat fallback behavior -> exposed two separate issues in backend defaulting and CLI fallback selection.
- Final approach: Make backend distinguish `ports=None` from `ports=[]`, and make CLI fallback probe only the remaining current ports -> eliminated duplicate probes and matched intended two-stage behavior.

## Lessons / Refinements
- Defaulting optional lists with `ports or ...` is unsafe when an empty list is a meaningful command.
- Multi-stage scan flows should track which ports have already been attempted before scheduling fallback work.
